MoQ C4M Auth Demo

Architecture and flow for Common Access Token authorization with live video over MOQT

◆ Components

Browser App
auth-demo (this page)
Orchestrates auth flow, publishes/subscribes video via MOQT, displays tokens and timeline
Google IDP
accounts.google.com
Identity Provider. Proves who the user is. Issues Google ID tokens (RS256 JWT)
moat (Token Service)
api.mocha-net.dev/v1
Validates Google ID tokens, creates sessions, mints ES256-signed C4M tokens with MOQT-specific scopes
moqx Relay
suhas-build-vm:443/moq-relay
MoQ relay. Verifies C4M tokens using moat's ES256 public key. Enforces per-namespace publish/subscribe authorization

◆ C4M Token Anatomy

// Header
{ "alg": "ES256", "typ": "CAT" }

// Payload
{
  "iss": "https://api.mocha-net.dev",
  "aud": ["moq-relay"],
  "sub": "user-id-from-google",
  "iat": 1751414400,
  "exp": 1751418000,
  "moqt": [{
    "actions": ["publish", "subscribe"],
    "namespace": "mocha/auth-demo"
  }]
}

// Signature: ECDSA P-256 (ES256) signed by moat's private key
alg: ES256 Asymmetric signature — relay verifies with moat's public key (no shared secret) moqt.actions What this token allows: publish, subscribe, or both moqt.namespace Which namespace/room the token grants access to exp Token expiry — relay rejects if current time > exp signature Relay verifies against moat's ES256 public key (pre-configured)

◆ Authorized Flow (Publisher & Subscriber)

Full flow: Google IDP → moat session → C4M token → relay accepts
Both publisher and subscriber follow this same path
1
Google Sign-In
Browser → Google IDP
User authenticates with Google. Browser receives a Google ID token (JWT signed by Google's RS256 key proving identity).
2
Exchange for moat session
POST /auth/google → moat
moat validates the Google ID token, creates a user record, returns a session_token. This proves identity to moat for subsequent API calls.
3
Mint C4M token
POST /token → moat (with session_token)
moat creates an ES256-signed C4M token with specific scopes: publish or subscribe on namespace mocha/auth-demo. This token is what the relay will verify.
4
Connect to relay (WebTransport)
Browser → QUIC CONNECT → relay
Opens a WebTransport session over QUIC/TLS. This layer does NOT check auth — any client can connect.
5
CLIENT_SETUP with C4M token
CLIENT_SETUP { token_type: 0x63346d, token: <ES256 JWT> }
The MOQT setup message carries the C4M token. The relay:
• Decodes the JWT
• Verifies ES256 signature against moat's public key ✓
• Checks exp not expired ✓
• Stores granted scopes for this session
6
SERVER_SETUP — Authorized!
relay → SERVER_SETUP
Relay accepts the session. Client can now PUBLISH or SUBSCRIBE within the granted namespace.
7
Publish / Subscribe video
Publisher: Encodes camera → sends video objects on [mocha/auth-demo]/video
Subscriber: Subscribes to same track → relay forwards frames → video plays

◆ Denied Flow (Invalid Token)

No IDP sign-in — uses a fake/invalid token
Demonstrates what happens when authorization fails
1
Generate invalid token locally
No Google sign-in. Token is crafted in the browser with intentional flaws (wrong scope, expired, bad signature, etc.)
2
Connect to relay (WebTransport)
Browser → QUIC CONNECT → relay
QUIC connection succeeds — transport layer doesn't check auth.
3
CLIENT_SETUP with invalid token
CLIENT_SETUP { token_type: 0x63346d, token: <INVALID> }
Relay attempts verification and FAILS:
• Wrong scope: namespace mocha/wrong-room ≠ requested resource
• Expired: exp is in the past
• Bad signature: doesn't verify with any known ES256 public key
• Guest/HS256: wrong algorithm (relay only has ES256 keys)
4
Connection terminated — DENIED
relay → closes connection / SUBSCRIBE_ERROR
Relay rejects the session or the specific subscribe action. No video is received.

◆ Authorized vs. Denied

Authorized (Panels 1 & 2)
  • Google IDP sign-in proves identity
  • moat mints ES256-signed C4M token
  • Token scoped to correct namespace
  • Relay verifies signature with moat's public key
  • SERVER_SETUP accepted
  • Video flows through relay
Denied (Panel 3)
  • No IDP sign-in (no identity proof)
  • Token crafted locally (fake signature)
  • Token scoped to wrong namespace / expired / bad sig
  • Relay verification FAILS
  • Connection rejected
  • No video received
Key Insight
Authorization happens at the MOQT protocol layer, not the transport layer. The QUIC/WebTransport connection succeeds for everyone — it's the C4M token verification inside CLIENT_SETUP that gates access. This means:

• Transport is open — anyone can establish a QUIC connection
• Protocol is gated — only valid C4M tokens unlock publish/subscribe
• Scopes are granular — a token can grant publish-only, subscribe-only, or specific namespaces
• Asymmetric crypto — relay needs only moat's public key (no shared secrets with clients)

◆ Token Type: 0x63346d

// In CLIENT_SETUP parameters:
AUTHORIZATION_TOKEN = {
  token_type: 0x63346d // ASCII "c4m" = Common Access Token for MoQ
  token: <base64url-encoded ES256 JWT>
}
The token_type field tells the relay how to interpret the token bytes. 0x63346d (decimal 6501485) is the registered type for C4M tokens, which signals the relay to decode it as a JWT/CWT with MOQT-specific claims in the moqt field.
